# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A pseudo-3D Outrun-style racing game built with plain HTML5 Canvas and vanilla JavaScript (no
frameworks, no build step, no package manager). It's a teaching project (by Jake Gordon,
jakesgordon.com) built up incrementally in four stages, each a fully self-contained HTML file:

- `v1.straight.html` — straight road only
- `v2.curves.html` — adds curves
- `v3.hills.html` — adds hills
- `v4.final.html` — final version: curves + hills + traffic cars + HUD + tweak UI + music

`index.html` is just a menu linking to the four versions.

The README explicitly states the code intentionally uses globals and embeds game logic directly
in `<script>` tags per HTML file rather than using modules — this is deliberate for
teachability, not an oversight. Don't "fix" it into a modular architecture unless asked.

## Running it

There is no build/install step. Serve the directory with any static file server and open the
HTML file in a browser, e.g.:

```
python3 -m http.server 8000
```

then visit `http://localhost:8000/v4.final.html` (opening via `file://` can break audio/image
loading in some browsers).

There is no test suite and no linter configured in this repo.

## Regenerating sprite sheets

`Rakefile` has a `resprite` task that rebuilds `images/sprites.js` / `images/background.js` (the
pixel-coordinate tables) from the individual PNGs in `images/sprites/` and `images/background/`,
using the Ruby `sprite_factory` gem:

```
rake resprite
```

You normally won't need this unless adding/replacing sprite images.

## Architecture

### Shared code (loaded by every version via `<script>` tags)

- `common.js` — everything shared across versions, organized into small object namespaces:
  - `Dom` — minimal DOM helpers (`get`, `set`, `on`, `show`, class toggling, `localStorage` wrapper as `Dom.storage`)
  - `Util` — math/helpers used throughout: interpolation/easing (`interpolate`, `easeIn`, `easeOut`, `easeInOut`), `accelerate`, `increase` (wrap-around increment), `overlap` (collision test), and `project` — the core pseudo-3D world→screen projection function
  - `Game` — the game loop and bootstrapping: `Game.run(options)` drives a fixed-timestep loop (accumulator pattern: `update(step)` called 0+ times per animation frame, then one `render()`), plus `loadImages`, `setKeyListener`, the FPS `stats` box, and `playMusic`
  - `Render` — all canvas drawing primitives: `polygon`, `segment` (draws one trapezoidal road segment incl. rumble strips/lane markers), `background` (parallax sky/hill/tree layers), `sprite`, `player`, `fog`
  - Constants: `KEY`, `COLORS` (road/grass/rumble/lane colors per segment type: light/dark/start/finish)
- `images/sprites.js` (`SPRITES`) and `images/background.js` (`BACKGROUND`) — generated pixel-coordinate lookup tables into `images/sprites.png` and `images/background.png` (the actual sprite sheets). Generated by `rake resprite`, not hand-edited.
- `stats.js` — third-party (mr.doob) FPS counter widget, used as-is.
- `common.css` — shared page styling for the HUD/tweak-UI controls.

### Per-version game code (inline `<script>` in each `v*.html`)

Each version defines its own tunable constants (road width, camera height/depth, field of view,
draw distance, fog density, speed/acceleration constants, etc.) and these key functions:

- **Road building**: `addSegment`, `addRoad`, and higher-level helpers (`addStraight`,
  `addHill`, `addCurve`, `addSCurves`, `addBumps`, `addLowRollingHills`, ...) compose a `segments`
  array — the entire track is just a flat array of segments with `curve`/`y` (height) values,
  looped via modulo indexing. `resetRoad()` (v3+) assembles the full track and places
  sprites/cars along it.
- **`update(dt)`** — game logic for one fixed timestep: reads input flags (`keyLeft`/`keyRight`/
  `keyFaster`/`keySlower`), updates `playerX`/`speed`/`position`, does sprite/car collision
  detection, updates lap timing.
- **`render()`** — walks `drawDistance` segments starting at the camera's current segment,
  projects each segment's two edges via `Util.project`, and calls `Render.segment` /
  `Render.sprite` / `Render.player` to draw the frame back-to-front.
- **`findSegment(z)`** — maps a world Z position to its segment via `Math.floor(z/segmentLength)`.
- **`reset(options)`** — (v4) applies tweak-UI option changes (resolution, lanes, road width,
  camera height, draw distance, field of view, fog density) and rebuilds the road only when a
  structural option (segment/rumble length) changed.
- Wiring happens at the bottom via `Game.run({ canvas, render, update, stats, step, images, keys, ready })`.

When editing gameplay/rendering, changes are almost always needed in exactly one `v*.html` file
(usually `v4.final.html`, the most complete version) plus, if the change is generic, in
`common.js`. The three earlier versions (`v1`–`v3`) are intentionally frozen snapshots of
progressively simpler states of the same game, kept for the accompanying tutorial articles — do
not "backport" v4 features into them unless explicitly asked.

## TypeScript Version (`app/`)

In addition to the original JavaScript version, this repository contains a faithful TypeScript port
in the `app/` directory. The TypeScript version preserves the original JavaScript code unchanged
as a reference and implements the same game logic using modern tooling (Vite, TypeScript strict
mode).

### Running the TypeScript version

```bash
cd app
mise exec -- npm install
mise exec -- npm run dev    # dev server
mise exec -- npm run build  # production build
mise exec -- npm run typecheck  # type checking
```

The TypeScript version follows the same incremental structure as the original:
- `v1.html` — straight road (TypeScript port of `v1.straight.html`)
- `v2.html` — curves (TypeScript port of `v2.curves.html`)
- `v3.html` — hills (TypeScript port of `v3.hills.html`)
- `v4.html` — final version with traffic, HUD, scenery (TypeScript port of `v4.final.html`)

### Architecture

The TypeScript version uses a class-based inheritance hierarchy (`RacerGame → RacerGameV1 → V2 → V3 → V4`)
following the Template Method pattern. Shared logic lives in `app/src/core/` (Dom, Util, Renderer, Road, RacerGame, TweakUI),
while version-specific extensions are in `app/src/versions/`.

For detailed architecture documentation, see `docs/projeto/01-arquitetura-alvo.md`.
